Session 1: Writing Custom Skills

Part 1: Understanding Skill Architecture (30 minutes)

What Makes a Good Skill?

A skill should be:

  • Reusable: Solves a problem you encounter repeatedly
  • Well-defined: Clear inputs and outputs
  • Composable: Can combine with other skills
  • Documented: Clear description and usage examples

Skill Components:

  1. Trigger: When should this skill activate?
  2. Instructions: What should Claude do?
  3. Context: What information does it need?
  4. Output: What format should the result take?

Example Skill Breakdown:

Skill Name: "Code Review Checklist"

Trigger: When reviewing a pull request or file changes

Instructions:

  • Check for security vulnerabilities (SQL injection, XSS, etc.)
  • Verify error handling and edge cases
  • Assess code readability and maintainability
  • Look for performance issues
  • Confirm tests exist and cover key paths

Context: The changed files and surrounding code

Output: Markdown checklist with findings and severity ratings

Part 2: Brainstorm Your Custom Skill (30 minutes)

Think about your workflow. What do you do repeatedly that could be automated?

Skill Ideas to Consider:

For Learning:

  • "Explain Like I'm a Beginner" - Adds detailed comments and explanations to complex code
  • "Generate Study Notes" - Summarizes a codebase or file into study material

For Development:

  • "API Endpoint Builder" - Scaffolds REST endpoints with error handling, validation, tests
  • "Database Migration Creator" - Generates migration files from schema changes
  • "Component Generator" - Creates React/Vue components with tests and stories

For Writing:

  • "Blog Post Drafter" - Converts notes into structured blog posts
  • "Technical Documentation" - Generates docs from code comments and function signatures
  • "README Generator" - Creates comprehensive READMEs from code analysis

For Productivity:

  • "Daily Standup Summary" - Summarizes your git commits into a standup update
  • "PR Description Writer" - Generates detailed PR descriptions from git diff
  • "Refactor Suggester" - Analyzes code and suggests refactoring opportunities

Choose One Skill to Build:

Pick something you'll actually use. Write out:

  1. The problem it solves
  2. When you'd use it
  3. What inputs it needs
  4. What output you expect

Part 3: Build & Test Your Skill (60-90 minutes)

Implementation Steps:

  1. Create the Skill File:
    • Follow Claude Code's skill format
    • Write clear trigger conditions
    • Document parameters and options
  2. Write the Skill Logic:
    • Define the instructions Claude should follow
    • Specify context gathering (which files, data sources)
    • Format the output template
  3. Add Examples:
    • Include sample inputs and expected outputs
    • Show edge cases
  4. Test Thoroughly:
    • Run the skill on real code
    • Try different scenarios
    • Refine based on results

Testing Your Skill:

Run your skill multiple times with different inputs:

  • Does it produce consistent output?
  • Are the results useful?
  • Does it handle edge cases gracefully?
  • Is it faster than doing the task manually?

Iteration is Expected

Your first version won't be perfect. Skills often need 3-5 iterations before they're truly useful. Pay attention to where they fail and refine the instructions.

Session 2: Multi-Agent Systems with Subagents

Part 1: Agent Orchestration Theory (30 minutes)

Why Use Multiple Agents?

Single agents can handle simple tasks, but complex workflows benefit from specialization:

  • Separation of Concerns: Each agent has a specific role
  • Expertise: Agents can have specialized knowledge (research vs. coding vs. writing)
  • Checks and Balances: One agent creates, another critiques
  • Parallel Processing: Multiple agents can work simultaneously

Common Orchestration Patterns:

1. Sequential Pipeline:

Agent 1 (Researcher) → Agent 2 (Writer) → Agent 3 (Editor)

Each agent passes its output to the next

2. Parallel + Synthesize:

Agent 1 (Option A) ⟍
Agent 2 (Option B) → Agent 4 (Synthesizer)
Agent 3 (Option C) ⟋

Multiple agents work independently, then a coordinator combines results

3. Iterative Refinement:

Agent 1 (Creator) ⇄ Agent 2 (Critic)

Two agents iterate back and forth until a quality threshold is met

4. Hierarchical:

Manager Agent → coordinates → Worker Agents

One agent delegates to others and synthesizes results

Part 2: Build a Multi-Agent System (90-120 minutes)

Now let's build a practical multi-agent system. You'll create a workflow where specialized subagents collaborate on a complex task.

Project: Research Report Generator

Build a system that produces a comprehensive research report on any technical topic.

Agent Roles:

  1. Researcher Agent:
    • Takes a topic and generates research questions
    • Searches for information (simulated or via APIs)
    • Compiles raw findings
  2. Analyst Agent:
    • Receives research findings
    • Identifies key themes and patterns
    • Structures information into an outline
  3. Writer Agent:
    • Takes the outline and raw research
    • Writes clear, engaging prose
    • Formats with proper citations
  4. Critic Agent:
    • Reviews the draft
    • Checks for logical gaps, unclear sections
    • Suggests improvements
  5. Editor Agent:
    • Incorporates critic feedback
    • Polishes language and formatting
    • Produces final report

Implementation Strategy:

  1. Define Agent Prompts: Write system prompts for each agent that define their role and constraints
  2. Create Communication Protocol: How do agents pass information? (JSON, structured text, etc.)
  3. Build the Orchestrator: The main script that coordinates agents
  4. Test Each Agent: Verify each works independently before connecting them
  5. Connect the Pipeline: Run the full workflow end-to-end

Test Your System:

Generate reports on topics like:

  • "The state of vector databases in 2026"
  • "Comparing frontend frameworks: React vs Vue vs Svelte"
  • "Best practices for API security"

Evaluate the output quality:

  • Is the research thorough?
  • Is the writing clear and well-structured?
  • Are the criticisms valid?
  • Does the final report feel cohesive?

Part 3: Reflection & Optimization (30 minutes)

Discussion Questions:

  • Where did the multi-agent approach excel vs. a single agent?
  • Which agent was the weakest link? How could you improve it?
  • What information got lost between agents?
  • If you had 10 agents instead of 5, what would you add?
  • How would you handle disagreements between agents?

Optimization Ideas:

  • Add Iteration: Let the Writer and Critic loop 2-3 times
  • Parallel Research: Run multiple Researcher agents on different aspects
  • Quality Gates: Don't proceed if an agent's output scores below a threshold
  • Human-in-the-Loop: Add approval steps between agents

Homework

Required Tasks

  1. Polish Your Custom Skill:
    • Test it on at least 5 different scenarios
    • Add comprehensive documentation
    • Create a README with usage examples
    • Consider publishing it for others to use
  2. Extend Your Multi-Agent System:
    • Add at least one more agent or improve an existing one
    • Implement error handling and retry logic
    • Generate 3 reports on different topics
    • Document the architecture with a diagram
  3. Write a Blog Post Draft:
    • Title: "Building a Multi-Agent System: Lessons Learned"
    • Explain the concept of agent orchestration
    • Share your architecture and results
    • Discuss what worked and what didn't
  4. Interview Prep:
    • Practice explaining your multi-agent system out loud
    • Prepare to draw the architecture on a whiteboard
    • Think through: "Why not use a single agent?"

Key Takeaways

  • Skills extend AI capabilities systematically - They're reusable, shareable, and composable
  • Agent orchestration is a design pattern - Applies beyond Claude to any multi-agent system
  • Specialization improves output quality - Dedicated agents outperform generalists on complex tasks
  • Communication between agents is critical - Clear protocols prevent information loss
  • Iteration and feedback loops matter - Critic agents and refinement passes significantly improve results
  • This is interview-worthy knowledge - Very few students understand agent orchestration at this level
