Pre-Work

None. This week is pure building. All your focus should be on implementation.

Build Week Philosophy

This is a working week, not a learning week. You have the skills—now apply them intensively. Expect to code 20-30 hours this week. Use Claude Code heavily, but stay in control. Your goal: by Sunday, your capstone should be substantially complete with all core features working.

Session 1: Working Session - Core Feature Implementation

Structure of Working Sessions

This week's "sessions" are focused build times with check-ins and guidance, not lectures.

Session Format (3-4 hours):

  1. Check-in (15 min): What are you working on today? Any blockers?
  2. Deep Work (2-3 hours): Focused implementation
  3. Demo & Review (30 min): Show progress, get feedback
  4. Planning (15 min): Set tasks for next session

Part 1: Leverage Your Week 5 Skills

Remember the custom Skills and Subagents you built in Week 5? Now's when they pay off.

Use Your Custom Skill:

  • If you built a code review skill, use it on your capstone code
  • If you built a documentation generator, use it for your README
  • If you built a test generator, use it to create test cases

Apply Multi-Agent Patterns:

For complex tasks, orchestrate subagents:

  • Implementer Agent: Writes the feature code
  • Tester Agent: Creates test cases
  • Reviewer Agent: Critiques the implementation
  • Documenter Agent: Writes docs for the feature

Prompt Strategy for Complex Features

"I need to implement [feature]. Use plan mode to:

  1. Design the implementation approach
  2. Identify edge cases
  3. Write the code with comprehensive error handling
  4. Create unit tests
  5. Update relevant documentation

Present the plan for approval before executing."

Part 2: Implementation Strategy

Feature-by-Feature Approach:

  1. Implement: Write the feature code
  2. Test Manually: Does it work in the happy path?
  3. Handle Errors: What breaks it? Add error handling
  4. Test Edge Cases: Try unusual inputs
  5. Write Tests: Automate your manual tests
  6. Document: Update README and docstrings
  7. Commit: Clear message, push to GitHub

When to Ask for Help:

  • Stuck for more than 30 minutes? Ask
  • Uncertain about architecture decision? Discuss
  • Claude giving unhelpful suggestions repeatedly? Get human input
  • Security-sensitive code? Get a review

Maintaining Momentum:

  • Pomodoro Technique: 50 minutes focused work, 10 minute break
  • Small Wins: Break features into tasks you can finish in 1-2 hours
  • Visible Progress: Update your GitHub issues/project board frequently
  • Avoid Rabbit Holes: If something is taking 3x longer than expected, reassess

Part 3: Quality Over Speed

It's tempting to rush. Don't. A polished 80% is better than a sloppy 100%.

Code Quality Checklist (for each feature):

  • Does it handle errors gracefully?
  • Are variable and function names clear?
  • Is complex logic commented?
  • Could another developer understand this?
  • Are there obvious bugs or security issues?
  • Is it tested?

When to Refactor:

  • If you're copy-pasting code, extract a function
  • If a function is > 50 lines, consider splitting it
  • If you find yourself confused by your own code, refactor now
  • If tests are hard to write, your code might be too complex

Technical Debt Is Real

Every shortcut you take now, you'll pay back with interest later. For a 3-week project, some debt is okay—but track it. Create GitHub issues for "known issues" and "future improvements" so interviewers see you understand the trade-offs.

Session 2: Open Source Contribution

Why Contribute Now?

You've been using AI tools all summer. Now give back. Contributing to the tools you use is both good karma and great for your resume.

Target Repositories:

  • AI Tooling: Claude Code, MCP servers, RAG frameworks, agent libraries
  • Type of Contributions:
    • Bug fixes (high impact, often easier than features)
    • Documentation improvements
    • Example code or tutorials
    • New features (if there's a clear need)

Finding an Issue to Work On (30 minutes)

Strategies:

  1. Check repos you've used: Have you hit bugs or limitations?
  2. Look for "good first issue" or "help wanted" labels
  3. Browse recent issues: Something from the last week might not be taken yet
  4. Documentation is always needed: Add missing examples, clarify confusing sections

Vetting an Issue:

  • Is the repo actively maintained? (recent commits, responsive maintainers)
  • Is the issue clearly described?
  • Can you reproduce the problem (for bugs)?
  • Is the scope reasonable (can you complete it this week)?

Claim the Issue:

  • Comment: "I'd like to work on this. My approach would be [brief description]. Does that sound right?"
  • Wait for maintainer response before investing significant time

Implementation (90-120 minutes)

Contribution Workflow:

  1. Fork the repo
  2. Create a branch: git checkout -b fix-issue-123
  3. Understand the codebase: Find relevant files, understand architecture
  4. Implement your fix/feature: Use Claude Code, but understand every change
  5. Test thoroughly: Run existing tests, add new ones if needed
  6. Follow style guide: Match the project's code style
  7. Commit with clear messages: Reference the issue number
  8. Push and create PR

PR Description Template:

Example PR Description

Title: Fix: Handle null values in data parser

Description:

"Fixes #123

Problem: The data parser crashed when encountering null values in the JSON response.

Solution: Added null checks before accessing nested properties. Falls back to default values.

Testing: Added unit tests for null, undefined, and missing properties. All existing tests pass.

Breaking Changes: None

Happy to make changes based on feedback!"

After Submitting:

  • Respond promptly to review feedback
  • Make requested changes quickly
  • Be gracious even if the PR isn't merged (it's their project, their call)
  • Learn from the review process

Homework

Required Tasks

  1. Implement All Core Features:
    • Every feature from your Week 9 spec should work
    • End-to-end testing of all major workflows
    • Error handling for common failure modes
    • No critical bugs
  2. Write Tests:
    • Unit tests for core functions
    • Integration tests for key workflows
    • Aim for >70% code coverage
    • Document how to run tests
  3. Complete Your OSS Contribution:
    • Submit your PR
    • Respond to any review feedback
    • Update your resume/LinkedIn with the contribution
  4. Update Documentation:
    • README reflects actual current state
    • Installation instructions work
    • Usage examples are accurate
    • Architecture doc is up-to-date
  5. Code Review with AI:
    • Use Claude Code to review your entire codebase
    • Fix any obvious issues it identifies
    • Document remaining known issues
  6. Daily Commits:
    • Commit at least once per day
    • Clear, descriptive commit messages
    • Push to GitHub regularly (backup!)

Productivity & Workflow Tips

Staying Focused During Build Week

Environment:

  • Remove distractions (close social media, turn off notifications)
  • Dedicated workspace
  • Good music or silence
  • Stay hydrated, eat well, sleep enough

Time Management:

  • Time-box features: "I'll spend 3 hours on this, then move on"
  • Track your time: Helps estimate remaining work
  • Schedule breaks: Your brain needs rest
  • End-of-day review: What did you accomplish? What's tomorrow's priority?

When You're Stuck:

  1. First 10 minutes: Read error messages carefully, Google the error
  2. Next 20 minutes: Try different approaches, add debugging prints
  3. After 30 minutes: Ask Claude Code / Claude for help
  4. After 60 minutes: Ask a human (mentor, peer, Stack Overflow)
  5. After 90 minutes: Take a break. Come back with fresh eyes.

Celebrating Progress:

  • Close an issue? Take a 5-minute break
  • Finish a major feature? Share a screenshot
  • Hit a milestone? Post on Twitter/LinkedIn
  • Momentum builds on visible progress

Key Takeaways

  • This is a marathon sprint - Intense but sustainable. Pace yourself.
  • AI tools are force multipliers - Use your Week 5 skills and subagents liberally
  • Quality compounds - Each well-built feature makes the next easier
  • OSS contributions are valuable - Even small PRs show you can work with real codebases
  • Testing isn't optional - Broken code is worse than missing features
  • Document as you go - You'll forget how things work by next week
  • Stay in control - Claude Code is your copilot, but you're the pilot
