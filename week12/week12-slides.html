Writing Effective Project Bullets

Do This

  • Start with action verbs (Built, Implemented, Designed)
  • Include metrics (200ms latency, 10k docs, 95% accuracy)
  • Highlight technical decisions
  • Explain challenges solved

Not This

  • Vague bullets ("worked on AI project")
  • No metrics or outcomes
  • Just listing technologies
  • Generic descriptions

Example

Built custom MCP server enabling Claude to search and analyze arXiv papers. Implemented PDF parsing pipeline processing 100+ papers with 95% accuracy. Designed resource handlers with caching, reducing API calls by 60%.

Answering Behavioral Questions

Situation

Set the context - what was happening?

Task

What was the goal or challenge?

Action

What did you do specifically? (Focus on YOUR actions)

Result

What was the outcome? Include metrics if possible.

Common Behavioral Questions

"Tell me about a time you got stuck on a hard problem", "Describe a technical decision you made and why", "What's a project you're proud of?"

8 / 12

Capstone Demo (5 min + Q&A)

1. Introduction (30 sec)

Name, project name, one-sentence description

2. Problem & Motivation (45 sec)

What problem does this solve? Why does it matter?

3. Demo (2-3 min)

Show the application working. Highlight 3-5 key features with real examples.

4. Technical Highlights (1 min)

Architecture overview, key technologies, interesting decisions

5. Lessons & Future (30 sec)

What you learned, what's next for the project

9 / 12

Presentation Best Practices

Do This

  • Practice timing (don't go over 5 min)
  • Speak clearly at moderate pace
  • Make eye contact
  • Show enthusiasm
  • Have demo ready to go

Avoid This

  • Staring at screen entire time
  • Apologizing or self-deprecating
  • Troubleshooting live
  • Rambling or going off-script
  • Showing broken features

Be prepared for: "What would you do differently?" and "What was the hardest part?"
