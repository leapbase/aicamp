Session 1: Build a Purpose-Driven Chatbot

Part 1: Choosing Your Chatbot's Purpose (20 minutes)

Generic chatbots are boring. Build one that solves a specific problem you care about.

Chatbot Ideas:

Study Buddy:

  • Helps with a specific class or subject
  • Explains concepts in simple terms
  • Generates practice problems
  • Uses flashcard-style interactions

Code Explainer:

  • Paste code, get line-by-line explanations
  • Identifies design patterns
  • Suggests improvements
  • Answers "why" questions about the code

Interview Prep Coach:

  • Asks technical interview questions
  • Provides hints when you're stuck
  • Evaluates your answers
  • Tracks progress over time

Personal Writing Assistant:

  • Helps brainstorm blog post ideas
  • Drafts outlines
  • Offers editing suggestions
  • Maintains your writing style

Recipe Adapter:

  • Modifies recipes based on dietary restrictions
  • Scales serving sizes
  • Suggests ingredient substitutions
  • Explains cooking techniques

Define Your Chatbot's Scope:

  • What specific problem does it solve?
  • Who is the target user?
  • What makes it better than ChatGPT?
  • What features are must-haves vs. nice-to-haves?

Part 2: Core Implementation (90 minutes)

Required Features:

1. Streaming Responses:

  • Display tokens as they arrive (not all at once)
  • Better perceived performance
  • User can stop generation mid-stream

2. Conversation History:

  • Maintain context across multiple turns
  • Store in memory (for now)
  • Allow users to start new conversations
  • Show conversation history in UI

3. Tool Use (at least one):

  • Add a function the chatbot can call
  • Examples: calculator, web search, database lookup, API call
  • Display when tools are being used

4. System Prompt:

  • Define your chatbot's personality and expertise
  • Set clear boundaries (what it will/won't do)
  • Include examples of good interactions

Technology Stack:

Backend: Python with Flask or FastAPI

Frontend: Streamlit (easiest) or simple HTML/JS

AI API: Claude or GPT-4 (your choice)

Implementation Steps:

  1. Backend Setup: API routes, conversation management, tool definitions
  2. Streaming Implementation: Server-sent events or websockets
  3. Tool Integration: Implement and test your function calling
  4. Frontend: Build the chat interface
  5. Polish: Error handling, loading states, clear buttons

Part 3: Add a Streamlit UI (45 minutes)

Streamlit makes it trivial to create web interfaces for Python apps.

Streamlit Basics:

  • st.chat_input() - User input field
  • st.chat_message() - Display messages
  • st.write_stream() - Streaming responses
  • st.session_state - Maintain conversation history

UI Enhancements:

  • Add a sidebar with settings or context
  • Display token usage and cost
  • Include a "New Chat" button
  • Show when tools are being invoked
  • Add example prompts to get users started

Test Your Chatbot:

Have real conversations with it:

  • Does it stay in character?
  • Does it handle edge cases gracefully?
  • Is the streaming smooth?
  • Do the tools work correctly?
  • Is it actually useful?

Session 2: RAG and Chat with Documents

Part 1: Understanding RAG (45 minutes)

The Problem RAG Solves:

LLMs have limitations:

  • Static Knowledge: Training data has a cutoff date
  • No Private Data: Can't know about your documents, codebase, or company data
  • Hallucination: Will confidently make up facts when it doesn't know

The RAG Solution:

  1. Retrieve: Find relevant documents or passages
  2. Augment: Add that context to the prompt
  3. Generate: LLM responds with grounded information

Without RAG:

User: "What did our Q3 earnings report say about cloud revenue?"

AI: "I don't have access to your specific earnings reports."

With RAG:

System: [Retrieves relevant section from Q3 report]

AI: "According to your Q3 earnings report, cloud revenue grew 34% year-over-year to $42M, exceeding expectations..."

How Retrieval Works: Embeddings

Embeddings are vector representations of text. Similar concepts are close together in vector space.

  • "dog" and "puppy" have similar embeddings
  • "dog" and "blockchain" have very different embeddings

The Process:

  1. Split documents into chunks
  2. Generate embeddings for each chunk
  3. Store embeddings in a vector database
  4. When user asks a question:
    • Embed the question
    • Find chunks with similar embeddings (cosine similarity)
    • Include those chunks in the prompt

Part 2: Build a "Chat with PDF" App (90-120 minutes)

Let's build a practical RAG application: upload a PDF, then chat with its contents.

Tech Stack:

  • Vector DB: Chroma (easy to use, no server needed)
  • Embeddings: OpenAI embeddings API or sentence-transformers
  • PDF Parsing: PyPDF2 or pdfplumber
  • UI: Streamlit
  • LLM: Claude or GPT-4

Implementation Steps:

Step 1: PDF Processing

  • Parse PDF into text
  • Split into chunks (500-1000 tokens each)
  • Handle overlapping chunks for context

Step 2: Create Embeddings

  • Generate embeddings for each chunk
  • Store in Chroma with metadata (page number, chunk ID)

Step 3: Query Interface

  • User asks a question
  • Embed the question
  • Retrieve top K similar chunks (K=3-5)
  • Show retrieved chunks in UI (for transparency)

Step 4: Augmented Generation

  • Build prompt: system instructions + retrieved chunks + user question
  • Call LLM API
  • Stream response to user

Step 5: Conversation Handling

  • Maintain conversation history
  • For follow-up questions, use original question + history for retrieval
  • Keep context window in mind (don't exceed token limits)

Example System Prompt for RAG:

"You are a helpful assistant that answers questions based on provided context. Follow these rules:

  • Only use information from the provided context
  • If the context doesn't contain the answer, say so explicitly
  • Cite page numbers when referencing specific information
  • If uncertain, express your level of confidence
  • Keep answers concise but complete"

Advanced Features (Optional):

  • Support multiple PDFs simultaneously
  • Highlight relevant passages in the original PDF
  • Allow re-ranking of retrieved chunks
  • Add hybrid search (keyword + semantic)
  • Cache embeddings to avoid re-processing

Part 3: Deploy Your Apps (45 minutes)

Time to make your chatbots accessible to others.

Deployment Options (All Free):

Streamlit Cloud:

  • Best for Streamlit apps (obviously)
  • Connect to GitHub repo
  • Automatic deploys on push
  • Free tier is generous

Vercel:

  • Great for Next.js or static sites
  • Serverless functions for backend
  • Fast edge network

Render:

  • Good for Python backends
  • Supports Docker
  • Free tier with auto-sleep

Deployment Checklist:

  1. Add requirements.txt with all dependencies
  2. Environment variables for API keys (never commit secrets!)
  3. Add error handling for production
  4. Set up logging for debugging
  5. Test thoroughly before making public
  6. Add rate limiting if needed
  7. Create a simple landing page explaining what your app does

Share Your Work:

  • Add deployment links to your GitHub README
  • Share on Twitter/LinkedIn
  • Ask friends to test and give feedback
  • Document any issues you encounter

  4. Experiment with Your RAG App:
    • Test with different types of PDFs (textbooks, papers, reports)
    • Try different chunk sizes and overlap settings

  5. Write a Technical README:
    • Explain your RAG implementation
    • Include architecture diagrams
    • Document design decisions (why Chroma? why these chunk sizes?)
    • Add troubleshooting section

Key Takeaways

  • Purpose-driven chatbots are more valuable - Generic assistants are commoditized; specialized tools stand out
  • Streaming improves perceived performance - Users prefer seeing incremental progress
  • RAG solves the knowledge limitation problem - LLMs + retrieval = grounded, factual responses
  • Embeddings enable semantic search - Vector similarity finds relevant content better than keywords alone
  • Chunk size matters - Too small loses context; too large wastes tokens and dilutes relevance
  • Deployment is part of development - If users can't access it, it doesn't exist
  • Real user feedback is invaluable - Your assumptions about UX are often wrong; test with real people
