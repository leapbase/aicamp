Capstone Philosophy

Your capstone should demonstrate mastery of what you've learned. It should be impressive enough to discuss in interviews, polished enough to put on your resume, and useful enough that you (or others) would actually use it. Quality over quantity—a well-executed, focused project beats an ambitious, half-finished one.

Session 1: Finalize Capstone Scope

Part 1: Project Direction Brainstorm (45 minutes)

Suggested Directions (Based on Camp Skills):

Direction 1: Custom MCP Server

Build a production-ready MCP server that solves a real problem.

Examples:

  • Academic paper search and summarization (arXiv + PDF processing)
  • Personal knowledge management (Notion/Obsidian connector)
  • Development environment manager (Docker + package managers)
  • Cloud resource manager (AWS/GCP connector with safe operations)

Why this direction: MCP is new and resume-valuable. A well-built server could be used by others.

Direction 2: Claude Code Skills Pack

Create a cohesive set of Skills for a specific domain.

Examples:

  • Frontend development pack (component generator, style optimizer, a11y checker)
  • Data science pack (EDA generator, visualization creator, model evaluator)
  • DevOps pack (deployment checker, config validator, monitoring setup)
  • Technical writing pack (documentation generator, style guide enforcer, example creator)

Why this direction: Shows deep understanding of a domain. Immediately useful to other developers.

Direction 3: Polished RAG Application

Build a production-grade RAG app with advanced features.

Examples:

  • Course material study assistant (upload syllabi, textbooks, notes)
  • Codebase documentation chat (index an OSS repo, answer architecture questions)
  • Research paper analyzer (upload papers, chat about methodology, findings)
  • Company knowledge base (internal docs, policies, FAQs)

Why this direction: RAG is highly practical. Good opportunity for thoughtful UX.

Direction 4: OpenClaw Skill Suite

Create a collection of powerful, integrated skills for OpenClaw.

Examples:

  • Personal productivity suite (calendar, tasks, email, notes)
  • Developer workflow automation (GitHub, CI/CD, documentation)
  • Content creation suite (research, draft, edit, publish)
  • Learning companion (flashcards, quizzes, spaced repetition)

Why this direction: Shows agent orchestration skills. Practical daily-use tool.

Direction 5: AI Feature for Open Source Project

Add substantial AI capability to an existing OSS project.

Examples:

  • Add AI-powered search to a docs site
  • Build an AI code review bot for a repo
  • Create AI-assisted issue triage system
  • Add natural language query to a data tool

Why this direction: Shows collaboration skills. Real-world impact if merged.

Choosing Your Direction:

Consider:

  • Your interests: What are you most excited about?
  • Skills showcase: What demonstrates your breadth best?
  • Usefulness: Would you or others actually use this?
  • Uniqueness: Does this differentiate you?
  • Feasibility: Can you complete it in 3 weeks?

Part 2: Scope Definition (60 minutes)

Once you've chosen a direction, define exactly what you'll build.

Project Specification Template:

  1. Project Name & Tagline: What is it in one sentence?
  2. Problem Statement: What problem does this solve? Who has this problem?
  3. Core Features (Must-Have): List 3-5 essential features. These define the MVP.
  4. Nice-to-Have Features: Additional features if time permits.
  5. Tech Stack: Languages, frameworks, APIs, databases.
  6. Success Criteria: How do you know it works? What does "done" look like?
  7. Non-Goals: Explicitly state what you won't build (prevents scope creep).

Example Specification:

Project: ArXiv Research Assistant MCP Server

Problem: Researchers spend hours manually searching arXiv for relevant papers and extracting key information.

Core Features:

  • Search arXiv by keywords, authors, categories
  • Download and parse PDF papers
  • Extract abstract, methodology, key findings
  • Summarize papers in plain language
  • Compare multiple papers on same topic

Nice-to-Have:

  • Citation network visualization
  • Save research collections
  • Alert on new papers matching criteria

Tech Stack: Python, MCP SDK, arXiv API, PyPDF2, Claude API

Success Criteria: Can find and summarize relevant papers 5x faster than manual search.

Non-Goals: Not building a paper recommendation system, not analyzing paper quality, not handling non-arXiv sources.

Validate Your Scope:

Ask yourself:

  • Can I describe this to someone in 30 seconds?
  • Can I realistically build this in 3 weeks?
  • Would I use this myself?
  • Does this showcase what I learned at camp?
  • Is this impressive but not overwhelming?

Session 2: Architecture & Setup

Part 1: Architecture Design (60 minutes)

Create Your Architecture Document:

1. System Overview Diagram:

  • Draw the major components
  • Show data flow
  • Identify external dependencies

2. Component Breakdown:

For each major component, document:

  • Purpose and responsibilities
  • Input and output
  • Dependencies
  • Key functions or classes

3. Data Models:

  • What data structures will you use?
  • Database schema (if applicable)
  • API contracts

4. External Integrations:

  • List all APIs you'll call
  • Authentication methods
  • Rate limits and costs
  • Fallback strategies

5. Error Handling Strategy:

  • What can go wrong?
  • How will you handle failures?
  • Retry logic
  • User-facing error messages

Why Architecture Matters

In interviews, you'll be asked to explain your capstone's design. A clear architecture shows engineering maturity. It also prevents you from getting stuck mid-build because you didn't think through a critical integration.

Part 2: Repository Setup (45 minutes)

Create Professional Project Structure:

Essential Files:

  • README.md - Comprehensive overview (start early, update often)
  • ARCHITECTURE.md - Your design document
  • requirements.txt or package.json - Dependencies
  • .gitignore - Don't commit secrets or build artifacts
  • .env.example - Template for environment variables
  • LICENSE - Choose an open source license (MIT is simple)
  • CONTRIBUTING.md (optional) - If you want contributions

Directory Structure Example:

my-capstone/
├── src/ (or lib/)
│ ├── core/ (main logic)
│ ├── api/ (external integrations)
│ ├── utils/ (helpers)
│ └── config/ (configuration)
├── tests/
├── docs/ (additional documentation)
├── examples/ (usage examples)
├── scripts/ (setup, deployment)
├── .github/workflows/ (CI/CD)
└── README.md

Set Up Development Workflow:

  • Version Control: Initialize git, create GitHub repo
  • Branches: Work on feature branches, merge to main
  • Commits: Write clear commit messages
  • Issues: Create GitHub issues for each major task
  • CI/CD: Set up basic GitHub Actions (linting, tests)

Part 3: Milestone Planning (30 minutes)

Define Weekly Milestones:

Week 9 (This Week):

  • Complete architecture and setup
  • Implement core data structures
  • Get one end-to-end flow working (minimal but functional)

Week 10:

  • Implement all core features
  • Add error handling
  • Begin testing
  • Start documentation

Week 11:

  • Polish features
  • Complete testing
  • Finish documentation
  • Deploy (if applicable)
  • Create demo video

Break Down Week 9 Tasks:

Create specific, actionable tasks for the next 7 days:

  1. Set up project structure
  2. Implement [specific component]
  3. Integrate [specific API]
  4. Build minimal UI/CLI
  5. Get one use case working end-to-end

Estimate hours for each task. Be realistic.

Homework

Required Tasks

  1. Finalize and Document Your Spec:
    • Complete project specification
    • Create architecture document with diagrams
    • Get feedback from mentor/peer
    • Refine based on feedback
  2. Set Up Repository:
    • Create GitHub repo with proper structure
    • Add all essential files
    • Write initial README (overview, setup, roadmap)
    • Create GitHub issues for all major tasks
  3. Build Core Foundation:
    • Implement basic project structure
    • Set up configuration and dependencies
    • Create core data models
    • Get one simple flow working (even if incomplete)
  4. Test Key Integration:
    • If using an external API, test it works
    • Verify authentication
    • Check rate limits and costs
    • Implement basic error handling
  5. Daily Progress Updates:
    • Commit daily with clear messages
    • Close GitHub issues as you complete tasks
    • Update project board or milestones
    • Document blockers immediately

Key Takeaways

  • Scope is critical - Better to nail a focused project than half-finish an ambitious one
  • Architecture prevents chaos - An hour of planning saves days of refactoring
  • Start simple, iterate - Get one thing working end-to-end, then expand
  • Documentation isn't optional - Start README and architecture docs now, not at the end
  • Milestones create momentum - Clear weekly goals keep you on track
  • Tools amplify, don't replace - Use Claude Code heavily, but understand what you're building
  • This is your portfolio piece - Quality and polish matter. Make it showcase-worthy.

Looking Ahead to Week 10

Next week is pure build time. You'll have working sessions and focused development. You'll also make your first open-source contribution during the camp (beyond Week 3). The goal: by end of Week 10, your capstone should be 80% feature-complete.
