What Makes a Great Capstone?

Quality Over Quantity

Your capstone should demonstrate mastery of what you've learned. It should be impressive enough to discuss in interviews, polished enough to put on your resume, and useful enough that you (or others) would actually use it.

A well-executed, focused project beats an ambitious, half-finished one

How to Choose Your Direction

Key Questions

  • What are you most excited about?
  • What demonstrates your breadth best?
  • Would you or others use this?
  • Does this differentiate you?
  • Can you complete it in 3 weeks?

Success Factors

  • Clear problem statement
  • Defined scope
  • Realistic timeline
  • Technical showcase
  • Portfolio-worthy polish
5 / 12

Define Your Scope

1. Project Name & Tagline

What is it in one sentence?

2. Problem Statement

What problem does this solve? Who has this problem?

3. Core Features (Must-Have)

List 3-5 essential features that define the MVP

4. Nice-to-Have Features

Additional features if time permits

5. Tech Stack & Success Criteria

Languages, frameworks, APIs. How do you know it works?

6. Non-Goals

Explicitly state what you won't build to prevent scope creep

6 / 12

Architecture Design

What to Document

  • System overview diagram
  • Component breakdown
  • Data models
  • External integrations
  • Error handling strategy

Why It Matters

  • Prevents mid-build chaos
  • Shows engineering maturity
  • Essential for interviews
  • Guides implementation
  • Identifies gaps early

An Hour of Planning Saves Days of Refactoring

In interviews, you'll be asked to explain your capstone's design. A clear architecture shows you think before you code.

Professional Project Structure

Essential Files

  • README.md
  • ARCHITECTURE.md
  • requirements.txt
  • .gitignore
  • .env.example
  • LICENSE

Directory Structure

  • src/ (main logic)
  • tests/
  • docs/
  • examples/
  • scripts/
  • .github/workflows/

Set Up Development Workflow

Git branching strategy, clear commit messages, GitHub issues for tasks, basic CI/CD with GitHub Actions

9 / 12

Implementation Philosophy

Start Simple, Iterate

Get one thing working end-to-end, then expand

Week 9 Goals

  • Project structure set up
  • Core data models implemented
  • One feature working fully
  • Documentation started

Avoid These Traps

  • Perfectionism (80% done > 100% perfect)
  • Scope creep (stick to spec)
  • Premature optimization
  • Skipping documentation
10 / 12

Homework: Plan & Build Foundation

  • Finalize and Document Spec: Complete project specification, create architecture document with diagrams, get feedback
  • Set Up Repository: Create GitHub repo with proper structure, add all essential files, write initial README
  • Build Core Foundation: Implement basic structure, set up dependencies, create core data models
  • Test Key Integration: If using external API, test it works, verify authentication, check rate limits
  • Daily Progress Updates: Commit daily with clear messages, close GitHub issues as you complete tasks
